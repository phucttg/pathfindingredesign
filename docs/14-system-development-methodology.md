# 3.2 System Development Methodology

## 3.2.1 System Objectives and Requirements

The development methodology was guided by both educational and technical requirements, reflecting the project's dual nature as both a learning tool and a software system. From an educational perspective, the primary objective was to make pathfinding algorithms less opaque by helping learners understand not only what an algorithm does, but why it selects certain nodes, prioritises particular routes, and produces a final path. This led to a requirement for step-by-step explanatory support, learner control over pacing, and visual features that promoted active exploration rather than passive observation. From a functional perspective, the system had to support a grid-based environment in which users could place start and target nodes, create walls, assign weights, generate maze layouts, select between algorithms, and visualise algorithm behaviour through animation. The system also needed to provide an explanation layer via the Insight Panel and a replay mechanism via the stored run history. In addition, non-functional requirements were treated as equally important. The tool needed to remain responsive during repeated interaction, support stable execution across multiple runs, and be lightweight enough for practical browser-based deployment. Since the system also included an optional AI-supported explanation service, it further required safe request handling, predictable fallback behaviour, and appropriate technical safeguards. These requirements ensured that the artefact was not only feature-complete, but also educationally purposeful, maintainable, and robust enough to meet the expectations of a computing-focused project.

## 3.2.2 High-Level System Architecture

The system follows a client-heavy architecture in which the core pathfinding and visualisation logic is executed in the browser. There is only a thin Express-based backend and a Vanilla-based front for the client side. The codebase includes a `POST /api/explain` endpoint that processes client requests and returns detailed insights required for the educational artefact. In this approach, the system avoids unnecessary server load and significantly reduces interaction latency, while keeping the main computation on the client. The backend, therefore, remained narrow in scope. The backend mainly serves static content, processes explanation requests, and supports optional AI-generated summaries with fallback behaviour when needed. Since the application was designed for single-user use, the web browser's `localStorage` was used to store run history and sidebar state, avoiding the complexity of implementing a database. Overall, the high-level architecture supported the project's educational goals while remaining technically efficient, maintainable, and well-suited to lightweight interactive web systems.

## 3.2.3 Client-Side Modules

Regarding the client-side aspects, several modules have been implemented across the client-side workflow. The board is a core component of the application and handles user interaction, customisation, and the state of algorithm execution. There are 8 distinct pathfinding algorithms implemented on the client side. Weighted and unweighted algorithm searches are handled through the board, and it also attempts to keep some of the underlying domain logic separate from presentation-related concerns.

Regarding state updates during execution, the animation engine here manages the visual playback of the search process and reconstructs the final path. Therefore, it ensures that the analysis-related elements, namely node updates, colour changes, performance metrics, and explanatory messages, remain aligned with the Insight Panel.

In the case of the explanation layer specifically, an AI-assisted summary feature has been implemented to turn run data into accessible text through a server-side OpenAI API call. Therefore, the system's output to the user can be much more accessible, rather than requiring a heavy technical background.

The history module in this pathfinding website is based on localStorage persistence and is included to support replaying and viewing previous runs for easier review. In addition, maze-generation utilities allowed users to construct more structured scenarios through which algorithm behaviour could be examined more systematically. The module generated these layouts by automatically placing walls or weighted nodes on the grid, using either structured methods such as recursive division or simpler random and predefined patterns.

## 3.2.4 Algorithm Implementation Strategy

Regarding the implementation of the search layer, the eight user-facing algorithms were not developed as eight fully separate program structures. Instead, they were organised into a smaller number of shared execution strategies within the browser. In this approach, the board acted as the client-side dispatcher and directed the selected algorithm into the appropriate search routine. Dijkstra, Greedy, Swarm, and Convergent Swarm were handled through a common weighted-search structure, while A* followed its own weighted routine. BFS and DFS shared an unweighted traversal structure, whereas Bidirectional search was implemented through a dedicated procedure in which two frontiers expanded simultaneously. Therefore, the implementation reduced unnecessary duplication while still preserving the distinct search behaviour required for each algorithm. Rather than returning a detached result object, these routines operated directly on a shared node graph so that execution state remained tightly connected to the grid visible to the learner.

At the node level, the search process relied on a shared state model in which each cell stored status, weight, predecessor links, and the cost values needed during traversal. Weighted algorithms updated these costs and predecessor references directly on node state as the search progressed. In this methodology, the cost model did not rely only on simple step distance, but also incorporated a base movement cost, turn penalties, and additional node weight when weighted cells were crossed. Dijkstra selected nodes according to accumulated cost, A* combined cost-so-far with a heuristic estimate, and Greedy relied more strongly on heuristic guidance than on full path cost. Swarm and Convergent Swarm further increased this heuristic bias so that the search moved more aggressively toward the target. By contrast, BFS and DFS were implemented as unweighted traversals in which the main distinction lay in the traversal structure itself, namely queue-like expansion for BFS and stack-like expansion for DFS. As a result, weighted and unweighted behaviour remained clearly differentiated in implementation terms.

In addition, the algorithms were instrumented not only to find a path, but also to support explanation and replay. During execution, they produced both a visited-node sequence and an execution trace that recorded key decision points, including current-node selection, neighbor evaluation, improved neighbor updates, skipped candidates, and target or midpoint discovery. The animation layer then used node-state changes and predecessor links to render the search process and reconstruct the final path, while the Insight Panel used the same trace to present live metrics, contextual explanations, and post-run reasoning. Replay was handled differently from simple trace playback. Rather than storing the full animation sequence, the history layer stored the board configuration, wall and weight placement, and algorithm settings, after which the selected algorithm was rerun to reproduce the behaviour. This shared execution pipeline therefore supported visualisation, explanation, and replay together, making the implementation strategy well aligned with the educational purpose of the artefact.

## 3.2.5 Data Storage and State Management

The project used a two-layer state model consisting of in-memory runtime state and persistent browser-side storage. During active use, the application maintained its primary state in memory so that node manipulation, algorithm execution, animation playback, and interface updates could occur with minimal delay. This in-memory layer stored information such as board configuration, node statuses, current algorithm settings, animation progress, and the state needed to drive the current visualisation. In contrast, persistent data was intentionally kept lightweight and limited to the browser's `localStorage`. Persistence was local only and was capped to a small recent-history set of five runs, together with limited interface preferences such as sidebar state. Rather than storing complete playback traces for later reproduction, the saved run data captured configuration details and summary outcomes; replay then reconstructed behaviour by re-executing the selected algorithm using the stored board state and settings. The choice of `localStorage` was appropriate because the stored information was educational rather than sensitive: it captured configurations and outcomes of visual runs rather than personal or confidential data. Run history therefore became a pedagogically useful feature, allowing learners to revisit and contrast different algorithms, weights, maze structures, and speeds. At the same time, keeping persistence local avoided the overhead and privacy implications of introducing a database or user account system. This state management strategy was therefore both technically efficient and methodologically aligned with the purpose of the tool, namely to support repeated, low-friction exploratory learning.

## 3.2.6 Reliability and Technical Safeguards

Reliability was treated as a core methodological consideration rather than a secondary implementation concern. Because the system was intended for repeated interactive use, it had to behave consistently under normal and error conditions. Several safeguards were therefore built into the application. First, interaction locking was used during active animation to prevent users from modifying the board in ways that could create inconsistent or partially updated states; in practice, this was implemented through button-locking behaviour during runs together with temporary restrictions on history interaction. Second, error handling was applied to invalid actions and interrupted runs so that the application could recover gracefully rather than fail silently or produce misleading output. Third, the explanation layer was designed with fallback behaviour so that the visualiser remained usable even if the AI explanation service was unavailable due to missing configuration or network failure. On the backend, technical safeguards focused on the single high-cost API route, where rate limiting reduced the risk of uncontrolled resource consumption and improved operational stability. Security-related HTTP headers and related hardening measures were also relevant because the tool used browser-side storage and dynamic interface behaviour. One important limitation is that neither the client-side explanation request flow nor the server-side AI call currently implements an explicit timeout or retry strategy, meaning that delayed or failed external responses are handled through fallback rather than robust request recovery. In addition, while synchronisation design reduced drift for the deterministic trace-based panel, AI summary timing can still vary, and delayed responses may overwrite the output of a later run in some edge cases. However, because the core explanatory logic was grounded in deterministic trace data rather than external generation alone, these limitations did not undermine the fundamental usability of the learning tool.

## 3.3 Key Implementation Decisions

### 3.3.1 Insight Panel and Trace-Based Explanations

The most important implementation decision in the project was to build the Insight Panel around trace-based explanation rather than around final-state summary alone. In conventional pathfinding visualisers, learners often observe coloured nodes and a final path without fully understanding the reasoning process that generated those outcomes. To address this problem, the system was designed to capture algorithm activity as a sequence of interpretable events. These events represented moments such as node expansion, frontier changes, weighted cost comparison, or heuristic prioritisation. This was especially important because different algorithms did not only produce different routes, but also selected nodes for different reasons: some prioritised accumulated cost, some combined cost with heuristic estimation, and others followed queue-based or stack-based traversal rules. Therefore, showing colour change alone would have made it difficult for learners to understand why a particular node was chosen at a particular stage. Instead of treating these events as internal program data only, the system mapped them to explanation text that described why a particular step occurred in language suitable for beginners. This design decision was grounded in the principle that educational visualisation is most effective when learners engage actively with the logic of the process rather than merely observe visual change. The Insight Panel therefore functioned as a bridge between algorithmic state and learner interpretation. It translated pathfinding behaviour into explanations that made decision-making visible, especially for users who may have found algorithm execution abstract or “black box” in more conventional learning materials. Where AI-generated summaries were available, they were deliberately positioned as supplementary rather than primary. The essential explanatory value of the system remained the deterministic event-to-explanation mapping derived from the run trace itself. This ensured that the core educational mechanism of the tool remained transparent, reproducible, and directly tied to the actual execution of the algorithm.

### 3.3.2 Animation Controller and Learner Control

A second major implementation decision was to treat animation as a learning mechanism rather than as a cosmetic feature. Pathfinding algorithms produce dynamic changes that can become difficult to follow when multiple nodes are explored rapidly. This was particularly relevant because some algorithms expand broadly and level-by-level, whereas others move in a more target-directed or depth-oriented manner, and these contrasting behaviours can become difficult to distinguish when shown too quickly. For this reason, the animation controller was designed to regulate pacing and improve cognitive manageability. The system supported different speed modes so that beginners could slow the process down to observe frontier growth and route selection in detail, while more confident users could accelerate playback or use an instant mode when they wanted to focus on outcomes. Where available, playback controls such as pausing, resuming, or step-wise progression further increased learner control over the visual flow of information. This decision was consistent with multimedia learning principles, particularly the idea that complex information is easier to process when it is segmented and paced in ways that users can control. Technically, the controller also played an important orchestration role by ensuring that visual updates, deterministic explanation text, and metrics remained aligned across the same sequence of events. This made replay more reliable and reduced explanatory drift for the trace-driven panel, although AI-generated summary timing could still vary independently of the main animation flow. In methodological terms, learner control was therefore embedded as both a usability feature and an instructional strategy designed to reduce overload and support closer attention to algorithm behaviour.

### 3.3.3 Weighted Nodes and Cost Representation

The implementation of weighted nodes was another important design decision because it enabled the tool to teach the conceptual difference between unweighted and weighted search more explicitly. In unweighted settings, learners often assume that the visually shortest route must also be the optimal one. However, weighted pathfinding requires the algorithm to optimise total cost rather than geometric distance alone. This distinction was especially important because learners could otherwise confuse the route that looked shortest on the grid with the route that was actually cheapest, particularly when comparing cost-aware searches with unweighted traversal behaviour. To support this distinction, the system allowed users to assign weights to cells and observe how these affected node selection and final path choice. This was especially important for algorithms such as Dijkstra’s algorithm and A*, where cumulative cost and, in the case of A*, heuristic guidance influence the search process. The user-facing representation of cost therefore needed to be educationally interpretable while remaining as close as possible to the underlying search logic. The interface was designed to make weight presence visible, while the explanation layer clarified why a path that appeared longer in shape could still be preferable because it had a lower total search cost. However, one technical limitation must be stated clearly: the search logic includes turn penalties in its internal evaluation, whereas the displayed and serialised UI path cost currently sums only base movement and node weight values. This means that the visible cost is useful for teaching weighted traversal, but it does not fully reflect every factor used by the search procedure. Acknowledging this distinction improves methodological accuracy and helps avoid overstating the precision of the cost visualisation from the learner’s perspective.

### 3.3.4 Run Serialisation and Replay

Run serialisation and replay were implemented as deliberate support for reflective learning and controlled comparison. Rather than limiting users to one transient animation at a time, the system captured each completed run as a serialised snapshot containing the board configuration, selected algorithm, speed setting, and structural changes such as walls and weights, together with summary outcomes from the run. These snapshots could then be stored locally, reloaded, replayed, or deleted. Importantly, replay did not depend on storing a full trace or complete playback sequence. Instead, runs stored configuration and summary outcomes, and replay reconstructed behaviour by re-executing the selected algorithm on the saved grid configuration and settings. From a technical perspective, this approach reduced storage overhead while still preserving enough information to recreate a meaningful learning experience. It also allowed the same saved configuration to be revisited under different algorithm choices, which supported controlled comparison between distinct search behaviours on the same grid. Educationally, rerunning the search was therefore more useful than preserving one fixed playback sequence, because it retained the exploratory value of comparison and iteration. The educational value nevertheless remained strong. Replay enabled users to compare how different algorithms behaved on the same grid, how the same algorithm reacted to different weights or maze layouts, and how pacing affected their understanding of the process. This supported an exploratory learning cycle in which users could revisit a run, modify one variable, and observe the consequences. It also strengthened methodological coherence between system design and research purpose, because the tool was intended to support understanding through interaction, iteration, and explanation. To remain professionally responsible, the stored data was designed to remain minimal and non-personal, and users retained control over deletion and clearing of stored runs. Replay therefore served as both a technical mechanism for lightweight state persistence and an educational mechanism for structured reflection.
